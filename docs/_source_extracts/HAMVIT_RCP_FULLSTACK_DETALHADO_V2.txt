 RCP FullStack e Arquitetura Técnica
 Detalhada
 Documento diretivo para implementação com Codex, Flutter, Supabase e ecossistema HAMVIT.
Este documento substitui a versão resumida. A finalidade é reduzir ambiguidades para o Codex e definir
decisões mandatórias de arquitetura, fluxo, regras de negócio, UX técnica, banco de dados, integrações
e critérios de aceite.

1. Regras absolutas para o Codex
Estas regras devem ser tratadas como obrigatorias. O Codex nao deve criar comportamentos, telas, campos, fluxos
ou estruturas fora do que estiver definido no RCP. Quando uma decisao nao estiver especificada, deve registrar
pendencia para decisao humana e nao implementar por conta propria.

Nao implementar WebView para o aplicativo mobile principal.

Nao adicionar anuncios, banners ou monetizacao baseada em publicidade.

Nao transformar o produto em assinatura mensal sem decisao explicita futura.

Nao bloquear funcionalidades basicas de saude no plano Free de forma agressiva.

Nao salvar dados sensiveis desnecessarios durante o onboarding inicial.

Nao permitir que IA registre calorias automaticamente sem revisao humana.

Nao usar selects longos sem busca/autocomplete.

Nao usar campos de texto livre para dados numericos, datas, CPF, CNPJ, telefone ou valores monetarios.

Nao expor segredos de Supabase service role, Mercado Pago, Resend ou IA no aplicativo Flutter.

Nao criar componentes visuais isolados fora do design system central.


A identidade visual deve ser premium, clean, moderna e acolhedora.



A navegacao principal deve usar bottom navigation fixa com no maximo cinco itens.

O menu hamburguer deve ficar no canto superior esquerdo e abrir drawer parcial com 75 a 85 por cento da


O drawer deve manter parte do conteudo ao fundo visivel, com animacao suave e suporte a swipe para fechar.

O drawer deve usar categorias expansivas com setas para cima/baixo, subitens contextuais e telas internas


Icones devem ser outline, elegantes, limpos e consistentes com a marca HAMVIT.

Evitar menus densos, telas poluidas, excesso de cards e excesso de metricas simultaneas.

6. Formularios, inputs e validacoes

Idade deve aceitar somente numero inteiro e teclado numerico.

Peso deve aceitar decimal, tratando virgula e ponto de forma consistente.

Altura deve priorizar centimetros inteiros para reduzir erro.

CPF deve usar mascara 000.000.000-00 e validacao de digitos quando usado.

CNPJ deve usar mascara 00.000.000/0000-00 e validacao quando usado.

Telefone deve usar mascara brasileira e validacao minima.

Data deve usar date picker nativo, nao campo livre.

Horario deve usar time picker nativo.

Campos monetarios devem usar mascara de moeda brasileira.

Codigos de cupom devem ser normalizados em uppercase.

Selects com listas extensas devem ser substituidos por autocomplete ou bottom sheet pesquisavel.
7. Autenticacao, perfis e coleta progressiva

Supabase Auth sera a autenticacao oficial.

Resend sera o provedor de e-mails transacionais com dominio proprio.

Cadastro do usuario comum deve ser leve, sem CPF ou endereco completo no onboarding inicial.

CPF, CNPJ, endereco ou dados fiscais so devem ser solicitados quando houver necessidade de pagamento,
fiscal, parceria ou comissao.

Nutricionistas/parceiros devem ter cadastro profissional inicial simples e dados financeiros apenas quando houver
comissao a receber.

A estrategia deve obedecer coleta progressiva de dados, LGPD, minimizacao de dados e baixa friccao.
Perfil
Dados iniciais permitidos
Usuario comum
Nome, e-mail, senha, objetivo, dados basicos de saude e preferencias iniciais.
Usuario Premium

Nutricionista/parceiro

Profissional com comissao
CPF/CNPJ, chave PIX/dados bancarios e dados fiscais somente quando necessario.

10. RLS, seguranca e privacidade

RLS deve estar ativo em todas as tabelas com dados pessoais ou dados de usuario.

Usuario comum so pode ler e alterar os proprios dados.

Profissional so pode acessar dados de paciente quando houver vinculo autorizado pelo usuario.

Admin deve acessar via painel com role especifica e auditoria obrigatoria.

Service role nunca deve ser usado no aplicativo Flutter.

Dados sensiveis locais devem ter protecao adequada conforme capacidade tecnica da plataforma.

Logs nao devem armazenar imagens, tokens, CPF completo, chaves PIX ou dados financeiros sensiveis sem
necessidade.

Operacoes administrativas devem gerar audit_logs.


O app deve permitir uso offline para habitos, agua, alimentacao manual, treino, peso, medidas, diario, historico
recente e captura de foto pendente.

Cada registro local deve possuir local_id, remote_id quando sincronizado, sync_status, created_at, updated_at e
last_sync_attempt_at.

A fila local deve tentar reenvio automatico quando a conexao voltar.

Erros temporarios devem gerar retry com intervalo progressivo.

Conflitos devem ser marcados como conflict e nao devem sobrescrever dados sem regra definida.

A interface deve mostrar estados claros de offline, sincronizando, sincronizado, pendente e falha.
Cenario
Regra de resolucao
Registro criado offline
Salvar localmente, marcar pending e sincronizar quando houver internet.
Atualizacao local sem conflito remoto
Enviar patch e marcar synced apos confirmacao.
Falha de rede
Manter pending ou failed, exibir mensagem amigavel e tentar novamente.
Conflito de edicao
Marcar conflict, preservar dados locais e exigir estrategia especifica de reconciliacao.
Foto de comida offline
Salvar localmente, marcar como pendente e processar IA somente ao retornar internet.
12. IA de foto da comida

Recurso exclusivo Premium, com limite inicial de 3 envios por dia por usuario.

Opcionalmente evoluir para 4 envios por dia mediante decisao futura.

O app deve comprimir imagem antes do upload e preservar qualidade suficiente para analise.

Backend deve receber imagem, validar permissao Premium, validar limite diario e chamar provedor de IA.

A IA deve retornar alimentos, porcoes estimadas, calorias, macros e nivel de confianca.

O usuario deve revisar e confirmar antes de salvar no diario alimentar.

A interface deve informar que valores sao estimados e ajustaveis.

A arquitetura deve permitir troca de provider sem mudar o app Flutter.
Provider
Uso definido
Gemini Flash Vision
Opcao inicial de melhor custo-beneficio para reconhecimento visual e JSON estruturado.
OpenAI Vision/GPT-4o mini
Alternativa de maior qualidade para testes ou fallback futuro.
Outros providers
Somente se integrados via camada abstrata de backend.
13. Scanner de codigo de barras

Scanner deve ser recurso tambem disponivel para usuarios Free por favorecer retencao e alimentacao diaria.

Flutter deve usar recurso nativo de camera/scanner.

Backend deve consultar primeiro a base local Supabase.

Se nao encontrar, consultar API externa como Open Food Facts.

Ao encontrar, salvar/cachear produto no Supabase com origem e status de confianca.

Usuario deve revisar informacoes e ajustar porcao antes de salvar.

14. Pagamentos e Premium

Modelo oficial: Free funcional + Premium vitalicio.

Gateway inicial: Mercado Pago.

Meios: PIX prioritario e cartao de credito opcional.





Premium deve desbloquear IA de foto, relatorios PDF, exportacoes, analytics avancados, historico avancado,
desafios e recursos premium.

Free deve manter valor real: habitos, agua, alimentacao manual, progresso basico, treino basico, calendario e
streaks simples.




Premium pode gerar PDF, compartilhar por WhatsApp, e-mail e share sheet nativo.
