 RCP Complementar V5 - Motor de Recomendacao Alimentar Premium
Objetivo: definir o modulo de sugestoes alimentares do HAMVIT, usando receitas curadas, metas diarias e consumo
registrado para sugerir refeicoes ao longo do dia.
Status: este documento complementa os RCPs FullStack, UX/Branding e Backend/Supabase. Ele nao substitui os
documentos anteriores. Deve ser anexado ao pacote final do Codex como modulo especifico de nutricao operacional.
Base oficial inicial: o arquivo HAMVIT_300_Receitas_Saudaveis.docx deve ser usado como fonte inicial para popular
o banco de receitas. Essa base contem receitas por categoria de refeicao, ingredientes, modo de preparo, objetivo
nutricional e informacoes estimadas de calorias, proteinas, carboidratos e gorduras.
1. Decisao de produto obrigatoria

O motor de recomendacao alimentar deve ser exclusivo para usuarios Premium.

Usuarios Free nao devem receber sugestoes automaticas de refeicoes, montagem do dia ou recomendacoes
baseadas em metas.

Usuarios Free podem registrar alimentacao manual, usar scanner de codigo de barras e visualizar progresso basico.

Premium deve desbloquear sugestoes alimentares, substituicao inteligente de receitas, montagem do dia, metas
restantes, insights e relatorios avancados.

As sugestoes devem ser tratadas como apoio de rotina, nao como prescricao nutricional individual.

4. Fluxo funcional do motor de recomendacao

O usuario Premium possui metas diarias de calorias e macros.

O app calcula o consumo ja registrado no dia em meal_logs e meal_items.

O backend calcula metas restantes por periodo e por refeicao.

O motor busca receitas da categoria correta: cafe da manha, almoco, jantar ou lanche.

O motor filtra receitas por calorias, proteinas, carboidratos, gorduras, preferencias, rejeicoes, favoritos, restricoes e
tempo de preparo.

O app exibe sugestoes em cards com nome, categoria, calorias, proteinas, tempo de preparo e botao adicionar ao
diario.

Ao aceitar uma sugestao, o usuario pode ajustar porcao antes de salvar.

Ao rejeitar uma sugestao, o sistema registra a rejeicao e evita repetir a mesma receita com alta frequencia.

A sugestao deve ser recalculada sempre que o usuario registrar nova refeicao, editar porcao ou mudar meta.

7. SQL complementar obrigatorio
create table public.recipe_meal_categories (
  id uuid primary key default gen_random_uuid(),
  key text not null unique,
  name text not null,
  sort_order integer not null default 0,
  active boolean not null default true,
  created_at timestamptz not null default now()
);
create table public.recipe_tags (
  id uuid primary key default gen_random_uuid(),
  key text not null unique,
  name text not null,
  active boolean not null default true,
  created_at timestamptz not null default now()
);
create table public.recipe_tag_links (
  id uuid primary key default gen_random_uuid(),
  recipe_id uuid not null references public.recipes(id) on delete cascade,
  tag_id uuid not null references public.recipe_tags(id) on delete cascade,
  created_at timestamptz not null default now(),
  unique(recipe_id, tag_id)
);
create table public.recipe_nutrition_profiles (
  id uuid primary key default gen_random_uuid(),
  recipe_id uuid not null unique references public.recipes(id) on delete cascade,
  calories_kcal numeric(10,2) not null,
  protein_g numeric(10,2) not null,
  carbs_g numeric(10,2) not null,
  fat_g numeric(10,2) not null,
  fiber_g numeric(10,2),
  sodium_mg numeric(10,2),
  serving_label text,
  serving_grams numeric(10,2),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
create trigger trg_recipe_nutrition_profiles_updated_at
before update on public.recipe_nutrition_profiles
for each row execute function public.set_updated_at();
create table public.recipe_seed_batches (
  id uuid primary key default gen_random_uuid(),
  source_file_name text not null,
  source_version text,
  imported_by uuid references auth.users(id) on delete set null,
  imported_at timestamptz not null default now(),
  total_records integer not null default 0,
  status text not null default 'pending',
  notes text
);
create table public.daily_nutrition_targets (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  target_date date not null,
  calories_kcal numeric(10,2) not null,
  protein_g numeric(10,2) not null,
  carbs_g numeric(10,2) not null,
  fat_g numeric(10,2) not null,
  water_ml integer,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  unique(user_id, target_date)

10. Algoritmo obrigatorio de recomendacao
O Codex deve implementar o motor com regras deterministicas inicialmente. Nao usar IA generativa para escolher
refeicoes na primeira versao. IA pode ser adicionada no futuro apenas como camada explicativa ou de refinamento.

Entrada: user_id, data, meal_type, metas diarias, consumo do dia, preferencias, rejeicoes, favoritos e receitas
disponiveis.

Calcular consumed_totals a partir de meal_logs e meal_items.

Calcular remaining_targets = daily_targets - consumed_totals.

Calcular target_for_meal usando meal_target_distribution ou distribuicao padrao.

Filtrar receitas por categoria de refeicao.

Excluir receitas marcadas como hidden ou dislike recente.

Priorizar favoritos quando encaixarem nas metas.

Pontuar por proximidade de calorias, proteinas, carboidratos e gorduras.

Penalizar receitas repetidas nos ultimos dias.

Retornar de 3 a 5 sugestoes por refeicao.

Salvar cada sugestao em user_meal_plan_suggestions com score e reason_json.
